## 15. Тарифы: рейсовый с ступенями по паллетам (Тип B — КВ/РД/Перспектива)
Тариф зависит от кол-ва паллет в рейсе: ≤8п→32 000, ≤10п→40 000, ≤15п→50 000, ≤18п→71 000.
Схема сейчас: один `pricePerTrip` без ступеней. Нужна модель тарифных ступеней:
`TariffTier { tariffId, maxPallets, pricePerTrip }` — тогда система сама выберет нужную строку.
Распределение: выручка = тариф_за_рейс ÷ паллет_всего × паллет_клиента (КВ + РД + Перспектива в одном рейсе).

## 16. Тарифы: ступенчатая ставка за паллет (Тип C — Поляна→Краснодар)
≤4 пал → 13 000 ₽/пал, ≥5 пал → 12 000 ₽/пал.
Аналогично п.15 нужна `TariffTier` с полем `minPallets / maxPallets` + `pricePerPallet`.
Или альтернатива — хранить как два отдельных тарифных ряда с пометкой в `notes`.

## 17. Тарифы: комбинированная ставка (Тип D — Поляна→Фреш Маркет)
паллет × 3 000 ₽ + 1 200 ₽ фикс за рейс.
В текущей схеме `pricePerPallet` и `pricePerTrip` оба могут быть заполнены — это уже работает
как комбинация. Нужно убедиться что логика расчёта выручки складывает оба компонента.
Также: Сферм→Фреш Маркет имеет особую ступень: до 5 пал фикс 9 000, далее +2 200/пал.

## 18. Тарифы: скомпилированный тариф для группы компаний
Несколько юрлиц одной группы (например КВ + Перспектива) ездят в одном рейсе и делят стоимость.
Нужен механизм «компилированного тарифа» — один тариф на маршрут, стоимость делится между
участниками группы пропорционально паллетам. Связано с п.19 (группы компаний).

## 19. Заявки: связь через родительский идентификатор (группы компаний)
Несколько заявок от разных юрлиц одной группы (КВ, РД, Перспектива) нужно объединять
под одним родительским ID. Это позволит: корректно считать скомпилированный тариф (п.18),
видеть все заявки группы в одном месте, правильно формировать счета.
Схема: добавить `parentRequestId` на `CustomerRequest` → самореференсная связь.

## 22. Счета: логика расчёта стоимости на основе типов тарифов
Текущая логика счётов не учитывает тип тарифа. Нужно реализовать расчёт суммы счёта:
- Тип A (за паллет): кол-во паллет × ставка
- Тип B (скомпилированный): рейсовая ставка ÷ паллет_всего × паллет_клиента
- Тип C (ступенчатый): выбор ставки по порогу паллет
- Тип D (ставка + фикс): паллеты × ставка + фикс
- Дополнительные услуги: суммируются сверху
Итог: система сама считает сумму счёта при его создании.
