# Правила работы Claude в проекте growfood-logistics

## Язык общения

- Отвечай по-русски по умолчанию.
- Будь краток и по делу, без воды и лишних объяснений.
- Давай критическую оценку запросов: если вижу проблему в подходе — скажи, не угождай.

## Выполнение shell-команд — КРИТИЧНО

Claude Code не может статически разобрать сложные shell-конструкции и в таких случаях **всегда запрашивает разрешение**, даже в режиме `bypassPermissions`. Чтобы этого избежать, **каждая команда должна быть простой и парсимой**.

### Запрещено использовать:

- `eval "$(...)"` и любые динамические подстановки внутри команды
- Heredoc (`<<EOF`, `<<'SQL'`, `<<-MARK` и т.п.)
- Многострочные цепочки через `&&`, `||`, `;` с переносами строк
- `node -e "..."` или `python -c "..."` с inline-скриптами длиннее одной строки
- Конструкции типа `cmd1 && cmd2 && cmd3` длиннее двух команд в цепочке

### Делай так:

- **Каждая команда — отдельный вызов tool.** Если нужно три действия — три вызова Bash подряд.
- **SQL → файл.** Вместо heredoc сохраняй SQL в `/tmp/query.sql` или `prisma/sql/<name>.sql`, потом `npx prisma db execute --file <path>`.
- **JS/TS → файл.** Вместо `node -e "..."` сохраняй в `scripts/<name>.js` и запускай `node scripts/<name>.js`.
- **Homebrew env не подключай.** Окружение уже должно быть в `~/.zshrc`. НЕ запускай `eval "$(/opt/homebrew/bin/brew shellenv)"` в командах.

### Пример: плохо vs хорошо

❌ Плохо (триггерит запрос разрешения):
```bash
eval "$(/opt/homebrew/bin/brew shellenv)" && cd /Users/vfedyanin/Documents/gftest && npx prisma db execute --stdin <<'SQL'
ALTER TABLE "Tariff" ADD CONSTRAINT ...;
SQL
```

✅ Хорошо (две простые команды):
```bash
echo 'ALTER TABLE "Tariff" ADD CONSTRAINT ...;' > /tmp/add_constraint.sql
```
```bash
npx prisma db execute --file /tmp/add_constraint.sql
```

## Работа с базой данных (Prisma)

- **Деструктивные операции — только с явного подтверждения пользователя в чате.** Это: `prisma db push --force-reset`, `migrate reset`, `db push --accept-data-loss`, `DROP TABLE`, `TRUNCATE`.
- Перед миграцией предлагай `prisma migrate dev --create-only`, чтобы пользователь сначала увидел SQL.
- Если БД может быть shared (dev/staging) — спрашивай, прежде чем что-то менять в схеме.
- Не запускай `prisma generate` сам после каждого изменения схемы — пользователь сделает это сам, когда нужно.

## Работа с файлами и git

- Перед массовыми изменениями файлов кратко опиши план (1–3 строки) и дождись согласия.
- Не делай "рефакторинг заодно" — меняй только то, о чём попросили.
- Не коммить сам — пользователь делает коммиты вручную.
- Если работа важная и git ещё не инициализирован — предложи `git init` и первый коммит **до** начала изменений.

## Контекст проекта

`growfood-logistics` — Next.js + Prisma проект для магистральной логистики GrowFood. Включает справочные и фактовые таблицы, поддержку LaaS-коммерции, многоостановочных рейсов, связанных заявок. Спецификация — `SPEC.md`, тест-кейсы — `TEST_CASES.md`.

## Скрипты для парсинга

Если задача связана с парсингом УПД или ОСВ 1С — используй сохранённые шаблоны (см. user memories). Для УПД из Google Drive — `read_file_content` (текст), не `download_file_content` (base64).
